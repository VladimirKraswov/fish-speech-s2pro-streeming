## Анализ архитектуры Fish Speech (fork streaming‑сервер)

### Общее назначение

Проект `fish-speech-s2pro-streeming` — это **серверная реализация синтеза речи (TTS)** на основе двухнейронной архитектуры:  
- **LLaMA‑подобная авторегрессионная модель (DualARTransformer)** генерирует **дискретные токены** (семантические + несколько уровней квантования).  
- **VQGAN‑декодер (DAC — Descript Audio Codec)** превращает эти токены обратно в **PCM‑аудио** (16‑бит, 44.1 кГц).  

Проект адаптирован под **RTX 5090 (CUDA 12.9, 32 ГБ VRAM)** и умеет работать как пакетно, так и в **стриминговом режиме** (аудио чанки по мере генерации).

## Структура папок (коротко)

```
fish_speech/                     # основной пакет движка
├── driver/                      # фасад для внешнего API
├── generation/                  # генерация токенов (Llama + семплирование)
├── inference_engine/            # оркестрация: Llama queue + декодирование
├── models/                      # реализации нейросетей
│   ├── dac/                     # VQGAN (кодек)
│   └── text2semantic/           # DualARTransformer + LoRA, KV‑кэш, квантизация
├── codec/                       # обёртка для VQManager (encode/decode токенов)
├── references/                  # загрузка и кэширование референсных голосов
├── tokenizer.py                 # токенизатор (текст → ID, семантические токены)
├── content_sequence.py          # построение промпта из мультимодальных частей
├── conversation.py              # высокоуровневая сборка диалога/промпта
└── utils/                       # вспомогательные утилиты

fish_speech_server/              # HTTP‑сервер (FastAPI‑подобный на Kui)
├── api/                         # эндпоинты: /v1/tts, /v1/health, /v1/references/...
├── proxy/                       # PCM‑прокси для браузерной потоковой передачи (NDJSON)
├── services/                    # ModelManager, адаптеры, обёртки инференса
└── schema.py                    # Pydantic‑схемы запросов/ответов

fish_speech_web_ui/              # Preact‑интерфейс (прокси + аудиоплеер)
docker/                          # Dockerfile (многостадийная сборка)
scripts/                         # скрипты up/down/status/deploy для RTX 5090
config/                          # runtime.json (пути к моделям, параметры)
```

> **Важно:** сам код инференса лежит в `fish_speech/`, а HTTP‑обёртка — в `fish_speech_server/`.  
> Внешний пользователь взаимодействует с эндпоинтами сервера (порт 8080) или с WebUI (9001).

## Ключевые компоненты (что за что отвечает)

### 1. **Driver** (`driver/api.py`, `driver/session.py`)
Предоставляет высокоуровневый класс `FishSpeechDriver`.  
Умеет:
- открывать сессии (`open_session`) для поддержки контекста (один голос на много запросов),
- синтезировать (`synthesize`) запросы,
- управлять референсами (добавление, удаление, список),
- выполнять прогрев (`warmup`) при компиляции модели.

### 2. **Inference Engine** (`inference_engine/__init__.py`)
Связывает Llama‑генератор и VQGAN‑декодер.  
- `TTSInferenceEngine` получает `DriverSynthesisRequest`,  
- через `ReferenceLoader` загружает аудиореференс (либо из кэша, либо кодирует на лету),  
- отправляет задание в потокобезопасную очередь `llama_queue` (запущенную в фоне),  
- получает сгенерированные токены,  
- вызывает `decode_vq_tokens()` для получения аудио‑чанков,  
- возвращает результат (токены или аудио) вызывающему коду.

### 3. **Генерация токенов** (`generation/`)
- `generation/worker.py` — фоновый поток, в котором запущена модель Llama.  
  Бесконечно читает `GenerateRequest` из очереди и вызывает `generate_committed_segments()`.
- `generate_committed_segments() (generation/prompt_builder.py)`  
  Строит промпт (системное сообщение + референс + текст), затем вызывает `generate()`.
- `generate() (generation/decode.py)`  
  Выполняет авторегрессивную генерацию с KV‑кэшем, семплированием (top‑p, temperature).  
  Поддерживает потоковую выдачу токенов (чанки по `stream_chunk_size`).
- `models/text2semantic/llama.py` — реализация **DualARTransformer**:  
  - первая часть (медленный трансформер) предсказывает **главный токен** (семантический или текст),  
  - вторая часть (быстрый трансформер) предсказывает **коды остальных `num_codebooks`** для этого же токена,  
  - используется `apply_rotary_emb`, `RMSNorm`, KV‑кэш, поддержка квантизации (int8/int4).

### 4. **Декодирование аудио** (`codec/`, `models/dac/`)
- `VQManager.decode_vq_tokens()` принимает тензор кодов формы `(num_codebooks, T)`  
  и вызывает `DAC.from_indices()`.
- Модель DAC (VQGAN) обучена на 44.1 кГц, 9 квантованных кодовых книг,  
  размер кода — 1024, семантическая книга — 4096.
- Декодер возвращает массив float32, который затем конвертируется в int16 PCM.

### 5. **Работа с референсами** (`references/loader.py`)
Сканирует папку `references/<id>/`, ожидая:
- либо `sample.wav + sample.lab` (кодируется при первом использовании),
- либо `voice.codes.pt + voice.lab` (предзакодированные токены, экономит время и память).  
Поддерживается кэширование в памяти (`use_memory_cache="on"`).

### 6. **HTTP API** (`fish_speech_server/api/views.py`)
- `/v1/tts` — основной эндпоинт. Принимает `ServeTTSRequest` (JSON или MessagePack).  
  Создаёт `DriverSynthesisRequest`, передаёт в `ModelManager.driver.synthesize()`.  
  Если `streaming=true` — отдаёт поток аудиочанков (RIFF WAV без заголовка после первого).  
  Если `streaming=false` — возвращает целый файл WAV.
- `/v1/health` — для Docker healthcheck.
- `/v1/debug/memory` — статистика GPU (выделенная/зарезервированная память).
- `/v1/references/*` — управление референсами (добавление, удаление, переименование, список).

### 7. **Прокси для браузера** (`proxy/pcm.py`)
Служит на порту 9000 и реализует **сессионный протокол**:
- клиент открывает сессию (`/session/open`), передаёт JSON‑конфиг (параметры TTS, коммит‑политику).
- затем отправляет порции текста (`/session/{id}/append`), сервер копит буфер и **коммитит** сегменты по правилам (точки, запятые, таймауты).
- для каждого коммита отправляется запрос в основной TTS (`UPSTREAM_TTS_URL`), а PCM‑данные возвращаются клиенту в виде NDJSON (каждый чанк — base64 PCM).  
Это позволяет браузеру плавно проигрывать синтезируемую речь.

## Полный flow: от текста до аудио

### Шаг 1. Пользователь отправляет HTTP‑запрос
```json
POST /v1/tts
{
  "text": "Привет, мир!",
  "reference_id": "voice",
  "streaming": true,
  "format": "wav",
  "max_new_tokens": 160,
  "top_p": 0.8,
  "temperature": 0.7,
  ...
}
```

### Шаг 2. API (`api/views.py::tts`)
- Валидирует запрос (наличие референса, длина текста, формат).
- Получает `ModelManager` из состояния приложения.
- Вызывает `driver.synthesize()` с `DriverSynthesisRequest`, сконвертированным через `api_tts_to_driver_request()`.

### Шаг 3. Driver (`driver/api.py::synthesize`)
- Открывает сессию (если не указано отдельно).
- Вызывает `engine.inference(request)`.

### Шаг 4. Inference Engine (`inference_engine/__init__.py::inference`)
- Через `ReferenceLoader` загружает токены референса (если `reference_id` указан).  
  При первом использовании аудио референса (WAV) он кодируется через `self.encode_reference()` (вызов VQGAN‑энкодера).
- Формирует параметры генерации: `max_new_tokens`, top_p, temperature, stream_tokens.
- Отправляет `GenerateRequest` в **очередь Llama** и ждёт ответа в другой очереди.

### Шаг 5. Фоновый worker (`generation/worker.py`)
- Извлекает запрос, вызывает `generate_committed_segments()`.
- **Сборка промпта** (`conversation.py` → `ContentSequence.encode_for_inference`):
  - Добавляется системное сообщение (например, "convert the provided text to speech").
  - Вставляются семантические токены референса (VQPart).
  - Добавляется пользовательский текст (текстовая часть).
  - Формируется массив `values` первой строкой (главные токены) и дополнительными строками (коды для каждого codebook).
  - Также генерируются `audio_masks` и `audio_parts` (если есть встроенное аудио, но в TTS не используются).
- **Генерация** (`generation/decode.py::generate`):
  - Модель получает prompt (`values`), начинает автогрессивно предсказывать следующие главные токены.
  - Для каждого главного токена **DualARTransformer** сначала вычисляет скрытое состояние через медленный трансформер, затем через быстрый трансформер предсказывает оставшиеся `num_codebooks` кодбуков.
  - Используется KV‑кэш, семплирование (top‑p, temperature).  
  - Если `stream_tokens = True`, то токены отдаются **чанками** (размер `initial_stream_chunk_size`, затем `stream_chunk_size`) через генератор.
- Полученные токены (тензор `(num_codebooks, T)`) оборачиваются в `GenerateResponse(action="sample", codes=...)` и отправляются в очередь ответа.

### Шаг 6. Inference Engine получает токены
- Для каждого `GenerateResponse` (чанка токенов) вызывает `self.get_audio_segment()`.
- `get_audio_segment()`:  
  - вызывает `self.decode_vq_tokens(codes)` (из `VQManager`),  
  - та, в свою очередь, вызывает `self.decoder_model.from_indices(codes[None])` (DAC декодирование),  
  - получает аудио‑массив `(samples,)` float32, переводит в numpy.
- Аудиосегмент упаковывается в `InferenceResult(code="segment", audio=(sample_rate, segment))`.
- Если `req.stream_audio = True` — каждый сегмент сразу же возвращается наружу (через `yield`).
- В конце возвращается `InferenceResult(code="final", audio=(sample_rate, full_audio))`.

### Шаг 7. Driver возвращает события
- Сессия `yield`'ит `DriverAudioChunkEvent` для каждого сегмента и `DriverFinalAudioEvent` в конце.

### Шаг 8. HTTP‑обработчик собирает ответ
- Если `streaming = True`:
  - Начинается `StreamResponse`. Для каждого `DriverAudioChunkEvent` генерируется чанк аудио (int16 PCM).  
  - Клиент получает поток **без заголовка WAV** (только голые PCM-данные).  
  - Браузер может воспроизводить их через Web Audio API.
- Если `streaming = False`:
  - Все чанки конкатенируются, к ним добавляется заголовок WAV (через `soundfile`),  
  - и весь файл отдаётся как один ответ.

### Шаг 9. (Дополнительно) Прокси‑сессии
- Для веб‑демо клиент подключается к `http://host:9000`.
- Сначала открывает сессию с конфигом, затем отправляет текст маленькими порциями.  
- Прокси сам решает, когда коммитить сегменты, и отдаёт PCM‑данные в NDJSON.

## Резюме

1. **Архитектура** — «нейронный кодек + авторегрессия»: LLaMA генерирует токены, DAC декодирует в аудио.
2. **Модульность**: драйвер → инференс‑движок → очередь генерации → декодер аудио. Всё распараллелено (отдельный поток для Llama).
3. **Потоковый режим**: токены отдаются чанками, аудио — тоже чанками, HTTP — `Transfer-Encoding: chunked`.
4. **Референсы** кэшируются на диске и в оперативной памяти (по желанию) — ускоряет первые запросы.
5. **Проект оптимизирован под 32 ГБ VRAM** (RTX 5090): используется `torch.compile`, int8/int4‑квантизация, настройки очистки кэша, контроль `max_new_tokens`.
6. **Код написан на чистом PyTorch** с минимальными зависимостями (`kui` for ASGI, `uvicorn`, `fastapi` для прокси).
